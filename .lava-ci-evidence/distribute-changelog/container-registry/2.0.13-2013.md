lava-api-go 2.0.13 (2013) — 2026-05-06

What's new vs 2.0.12 (2012):

FIXED:
- deployment/thinker/thinker-up.sh: lava-api-go-thinker now runs with
  `--network host` instead of a podman bridge network. The previous
  bridge setup confined mDNS broadcasts to the bridge subnet, so
  Android testers running Lava on the LAN could NOT auto-discover the
  API via `_lava-api._tcp`. Matches the local docker-compose.yml
  pattern where lava-api-go uses `network_mode: host` for the same
  reason. Postgres still uses a bridge network with port published
  to 127.0.0.1; api-go connects via 127.0.0.1:5432 from the host
  namespace.

Verification (post-fix):
  ssh thinker.local 'podman logs lava-api-go-thinker' shows:
    {"msg":"mDNS announced","port":8443,"type":"_lava-api._tcp"}
  curl https://thinker.local:8443/health from the workstation returns:
    {"status":"alive"}
  ss -tln on thinker.local shows:
    LISTEN  *:8443  (api-go on all interfaces, not bridge-confined)

REGRESSION CONTEXT:
- 2.0.11 introduced remote distribution to thinker.local but used a
  podman bridge network in the boot script. The bridge network
  confined mDNS — a §6.G End-to-End Provider Operational Verification
  miss because the LAN-discovery flow was not exercised end-to-end
  before claiming "API distributed and booted on thinker.local".
- 2.0.13 closes this gap. The §6.J Anti-Bluff observation: a healthy
  /health probe is necessary, never sufficient. mDNS reachability is
  part of the user-visible flow (Android client auto-discovery).
