lava-api-go 2.0.11 (2011) — 2026-05-06

What's new vs 2.0.10 (2010):

ADDED (remote distribution to thinker.local):
- New scripts/distribute-api-remote.sh — ships the Go API OCI image
  tarball + boot script + TLS material to a remote host via SSH and
  brings the stack up under rootless Podman. The remote host MUST be
  passwordless-SSH-reachable; default target is thinker.local.
- New deployment/thinker/{thinker.local.env, thinker-up.sh} — boot
  config + script that runs on the remote host, loading the image
  tarball, starting an isolated lava-postgres-thinker + lava-api-go-
  thinker pair on a dedicated podman network, and waiting for /health
  before reporting success.
- New docs/REMOTE-DISTRIBUTION.md — operator runbook for the remote
  flow (initial setup, distribute, verify, tear-down).
- .env.example documents LAVA_API_GO_REMOTE_HOST + LAVA_REMOTE_HOST_USER
  with safe defaults (thinker.local + milosvasic).

OPERATIONAL:
- Local-host Podman containers + image are torn down at the end of
  the distribute cycle to free the operator's workstation. Going
  forward, lava-api-go runs on thinker.local; the local workstation
  only builds. Android client testers reach the API via mDNS
  discovery (which already advertises any LAN host).

CONFIGURATION (per §6.H Credential Security):
- LAVA_API_GO_REMOTE_HOST / LAVA_REMOTE_HOST_USER come from .env
  (gitignored). The remote host's Postgres credentials live in
  deployment/thinker/thinker.local.env (also gitignored at deploy
  time; the canonical default is the operator-acceptable fallback).
- TLS material (server.crt + server.key) is copied into the remote's
  ~/lava/tls/ on each distribute. Production cutover should
  generate per-host certs.

CONSTITUTIONAL BINDINGS:
- §6.J Anti-Bluff — distribute-api-remote.sh propagates failures
  via set -euo pipefail; /health is verified end-to-end before
  reporting success.
- §6.B "Container Up" is not application-healthy — the boot script
  waits for /health on the remote, not just for podman ps to
  report Up.
- §6.M Host-Stability — rootless Podman; no host power-management
  commands.
- §6.K Builds-Inside-Containers — image is produced by
  build_and_release.sh which already routes through the project's
  container build path.
- §6.P Distribution Versioning + Changelog Mandate — this file IS
  the per-version snapshot; CHANGELOG.md gains a matching entry.
