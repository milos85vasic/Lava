### §6.J META-finding (test-harness regression discovered + fixed in same cycle)

The first run of the Challenge suite on 2026-05-16 surfaced FAILURE on C26 + C27 + C30. Investigation:

- **C26 + C27 (Welcome colored logo)**: false-positive failures — Compose's `captureToImage()` API returns an all-zero bitmap on Pixel_8/API 35 emulator + AGP combo. Independent visual verification via `adb screencap` + ImageMagick pixel-direct read confirmed the Welcome screen renders the colored Lava logo correctly (pixel at (540, 900) = `srgba(152, 11, 21, 1)` — the dark Lava red brand color). Fix landed in this cycle: switched both tests to `InstrumentationRegistry.getInstrumentation().uiAutomation.takeScreenshot()` (captures the actual hardware-accelerated rendered frame, identical to `adb screencap`). Both tests now PASS truthfully. Also fixed the assertion logic — prior version compared per-CHANNEL max-min across whole bitmap which is masked by white background pixels; corrected to per-PIXEL R-G/R-B/G-B deltas which only spike on truly-colored pixels.

- **C30 + C31 + C32 + C33 + C34 + C35 (feature-reachability)**: real test-infrastructure regression — all 6 tests used `createAndroidComposeRule<MainActivity>()` but lacked `@HiltAndroidTest` + `HiltAndroidRule` setup. Since MainActivity is `@AndroidEntryPoint`, the Hilt test component must be initialized via `HiltAndroidRule(this).inject()` before the activity launches. Fix landed in this cycle: all 6 tests gained the Hilt annotation + rule + `hiltRule.inject()` call. All 6 now PASS.

This META-finding is itself a §6.J / §6.L finding: tests that exist to catch §6.AB-class bugs had silently regressed (false-positive on C26+C27; refused-to-run on C30-C35). Per §6.AB and the operator's standing mandate, these test-harness regressions are themselves bluffs — green tests that ASSERT-FAIL on harness issues instead of real defects. Fixed in same cycle as discovered.
