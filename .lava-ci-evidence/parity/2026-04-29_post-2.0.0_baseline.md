## Remaining divergences вЂ” post-release polish

These do not block the 2.0.0 release; the Android client (SP-3) is the right place to decide whether to align both backends or to teach the client to handle either shape.

### A) JSON `null` vs omitted (cosmetic)

oapi-codegen-generated `CategoryDto.Children` and `CategoryDto.Id` are `*[]CategoryDto` / `*string` with the `,omitempty` JSON tag. When nil, Go emits nothing; Ktor's kotlinx-serialization emits `null`.

**Fix paths**:
- (a) Update `api/codegen-server.yaml` to use a custom template that drops `omitempty` for nullable fields, regenerate, re-commit `internal/gen/server/api.gen.go`.
- (b) Add a `MarshalJSON` method on a wrapper type in `handlers/` that always emits null for nullable-nil fields.
- (c) Add a JSON post-processor in `writeJSON` that injects `null` for missing fields the OpenAPI marks nullable.

(a) is most idiomatic; (c) is most localized.

### B) Anon-user semantic short-circuits (behavioral)

The Ktor proxy gates anon `GET /favorites`, `GET /search`, `GET /torrent/{id}` (with invalid id) and emits a 401/404/{} response without calling rutracker. The Go API forwards every request and returns whatever rutracker says (either 200 with empty data or a propagated 401/404 with an error body).

Both behaviors are wire-valid; the question is which the Android client prefers. Defer to SP-3 brainstorm.

### C) Content-Type for the binary download routes

Not exercised in this fixture set. Confirm `/captcha/{path}` and `/download/{id}` emit the upstream Content-Type byte-equal in the SP-3 acceptance run.

## Falsifiability rehearsal status

The 9 comparator unit tests in `parity_test.go` (TestCompareResponses_*, TestCanonicaliseJSON_*) pass unconditionally вЂ” they prove the comparator can distinguish all the failure modes the live test surfaced. The three plan-mandated rehearsals against real backends:

- (a) **corrupt body**: implicitly demonstrated вЂ” the live `get_forum_anon` failure shows `canonical JSON differs at offset 97` with the offending bytes, exactly the form a deliberate body corruption would surface.
- (b) **reorder JSON keys**: the canonicalise step removes order from the comparison. To rehearse: revert the canonicalisation (use `compare: exact`) on a fixture, observe key-order failure, restore. Not exercised on the live run because no backend has out-of-order keys today.
- (c) **drop a header**: not exercised this run вЂ” both backends emit the headers the fixtures' allowlist covers.

The full three-rehearsal protocol from the plan is mechanical to run when the cosmetic and semantic divergences are addressed.
